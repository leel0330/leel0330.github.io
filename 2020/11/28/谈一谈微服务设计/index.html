谈一谈微服务设计

发表于 2020-11-28 分类于 Tech 阅读次数：

现如今是微服务大行其道的时代，随着微服务大量被使用，使用者对其优点和缺点也有了更深一步的认识。为了能够合理利用微服务技术，需要遵守一些原则，规避微服务的缺点和风险。因此，这篇文章阐述微服务的一些特点，也会谈到该怎么样正确使用微服务，使它为我们的业务保驾护航。

本文章的内容主要来自于《微服务设计》这本书。

1. 微服务是什么？为啥要使用它？

1.1 什么是微服务

从字面上讲微服务就是小的服务，多小算小呢？代码库小就是微服务吗？微服务的微理解就是指一些协同工作的小而自治的服务。它主要有如下几个特点：

  • 很小，小到专注于做好一件事情。就像设计模式里学的，遵守单一职责原则。
  • 自治，一个微服务就是一个独立的实体，独立部署在PAAS上，或者作为一个进程存在。服务间如果有依赖，可以通过网络调用，增加服务间的隔离性，避免耦合。

1.2 微服务的好处

金坷垃我知道了，那好处都有啥？为啥要买金坷垃？嘻嘻，好处一一让我道来：

  • 技术异构性。白话就是不同微服务可以采用不同的技术，比如A服务用Go开发，使用TiDB数据库，而B服务用Java开发，使用MySQL数据库。相对于单体只能用一种语言开发，这样拆分出不同的服务，可以合理利用每种语言或基础服务的优点。
  • 弹性。白话就是随着业务的发展，规模逐渐扩大，这个时候需要使用更多的实例来撑起我们的业务，微服务可以很容易做到水平扩展。
  • 简化部署。单体里即使改一行代码也要部署整个应用，这种方式部署的影响很大，风险很高。而微服务只需要部署特定部分的代码，即使该服务出了问题，也不会对其他服务造成影响，也更容易进行回滚操作。
  • 对可替代性的优化。假使某个服务开始为了快速上线而使用了能够快节奏开发的语言。但是后来人员充裕，能够替换成性能更强、稳定性更高的语言，这个成本相对单体而言是比较低的。

1.3 微服务虽好，但其实没有银弹

需要强调一点，微服务不是免费的午餐，更不是银弹。适合的才是最好的。是否使用微服务，要考量很多因素。

2. 一步就位很难，服务一般都是演化而来的，因此需要演化式架构师

架构师必须改变那种从一开始就要设计出完美产品的想法，你猜的没错，架构师需要拥抱变化，应该设计出适用于当下的合理的框架，在这个框架下可以慢慢演化出正确的系统。架构师要着眼于当下，放眼于未来，在当下和未来之间做出一个很好的权衡。需要保证系统不但满足当前的需求，也应该能够应对将来的变化。

对于技术领导人来说，更重要的事情是帮助你的队友成长，帮助他们理解这个愿景，并保证他们可以积极地参与到愿景(俗称画大饼)的实现和调整中来。

一个演进式架构师应该承担的职责：

  • 愿景。确保在系统级有一个经过充分沟通的技术愿景，这个愿景可以帮助你满足客户和组织的需求。
  • 同理心。理解你所做的决定对客户和同事带来的影响。
  • 合作。和尽量多的同事进行沟通，从而更好地对愿景进行定义、修改和执行。
  • 适应性。确保在你的客户和组织需要的时候调整技术愿景。
  • 自治性。在标准化和团队自治之间寻找一个正确的平衡点。
  • 治理。确保系统按照技术愿景的要求实现。

成功要靠不断地取舍来实现。僵化固守自己的想法，刚愎自用无疑是最糟糕的做法。

3. 想法和架构师都有了，那该怎么建模服务呢？

在动手实践微服务之前，我们需要理解一点，就是标准是什么？什么样的服务才是好服务呢？好的服务不外乎以下几点:

  • 松耦合。如果做到松耦合了，修改一个服务就不需要修改另一个服务，能够独立修改以及部署单个服务而不需要修改其他系统的其他部分。
  • 高内聚。把相关的行为聚集在一起，把不相干的行为放在别处。所以，找到问题的边界就可以确保相关的行为能够放在一起，并且它们会和其他边界以尽量松耦合的形式进行通信。

这两个特点是不是很熟悉，学设计模式的时候也经常会强调这两点。

3.1 限界上下文

任何一个给定的领域都包含多个限界上下文，每个限界上下文中的东西分为两部分，一部分不需要对外部通信，另一部分则需要。每个上下文都有明确的接口，该接口决定了它会暴露那些模型给其他的上下文。

微服务应该清晰地和限界上下文保持一致。如果服务边界和领域的限界上下文能保持一致，并且微服务可以很好地表示这些限界上下文的话，就跨出了走向高内聚低耦合的微服务架构的第一步。

过早将一个系统划分为微服务的代价非常高，尤其是面对新领域时。很多时候，将一个已有的代码库划分为微服务，要比从头开始构建微服务要简单的多。

3.2 业务功能

在思考组织内的限界上下文时，不应该从共享数据的角度来考虑，而应该从这些上下文能够提供的功能来考虑。

3.3 逐步划分上下文

一开始你会识别出一些粗粒度的限界上下文，而这些限界上下文可能又包含一些嵌套的限界上下文。当考虑微服务的边界时，应该首先考虑比较大的、粗粒度的那些上下文，然后当发现合适的缝隙后，再进一步划分出那些嵌套的上下文。

3.4 业务概念的沟通

修改系统的目的是为了满足业务需求。

以跟组织内通信相同的方式来思考微服务之间的通信形式是非常有用的。

3.5 技术边界

技术边界不一定和限界上下文保持一致，因此，什么时候需要按照技术边界来划分是需要根据具体情况考虑的。

4. 集成

上一章讨论了什么是好的服务，并且知道要按照限界上下文来划分服务，使这些微服务以高内聚低耦合的方式运行着。但是现实情况是微服务之间或多或少的有一些联系，那该怎么操作呢？

4.1 寻找理想的集成技术

微服务之间通信的方式的选择非常多样化，但哪个是正确的呢？在选择之前，需要考虑我们到底希望从这些技术中得到什么。

  • 避免破坏性修改。
  • 保证API的技术无关性。不应该选择那种对微服务的具体实现技术有限制的集成方式。
  • 使你的服务易于消费方使用。
  • 隐藏内部实现细节。

4.2 需要考虑的技术点

4.2.1 共享数据库

永远避免共享数据库。

4.2.2 同步与异步

使用同步通信(请求/响应模式)，发起一个远程服务调用后，调用方会阻塞自己并等待整个操作的完成。

如果使用异步通信(基于事件模式)，调用方不需要等待操作完成就可以返回，甚至可以不需要关心这个操作完成与否。异步通信对于运行时间比较长的任务来说比较有用。

当你需要低延迟的时候，通常会使用异步通信，否则会由于阻塞而降低运行的速度。

4.2.3 编排与协同

编排方式的缺点会导致少量的上帝服务，而与其打交道的那些服务通常会沦为贫血的、基于CRUD的服务。

使用协同的方式可以降低系统的耦合度，并且你能更加灵活的对现有系统进行修改，但确实需要额外的工作来对业务流程做跨服务的监控。

4.2.4 远程过程调用(RPC)

当决定选用RPC的时候，需要注意一些问题：

  • 不要对远程调用过度抽象，以至于网络因素完全被隐藏起来。
  • 确保可以独立地升级服务端的接口而不用强迫客户端升级。
  • 在客户端中一定不要隐藏是在做网络调用这个事实
  • 在RPC的方式中经常会在客户端使用库，但是这些库如果在结构上组织的不够好，可能会带来一些问题。

4.2.5 REST

基于HTTP的REST是一个比较合理的默认选择。

4.2.6 实现基于事件的异步协作方式

尽量让中间件保持简单，而把业务逻辑放在自己的服务中。

事件驱动架构和异步编程会带来一定的复杂性，所以通常需要谨慎地选用这种技术。

4.2.7 服务即状态机

把关键领域的生命周期显式建模出来非常有用，不但可以在唯一的一个地方处理状态冲突，而且可以在这些状态变化的基础上封装一些行为。

4.2.8 响应式扩展

响应式扩展提供了一种机制，在这种机制之上，你可以把多个调用的结果组装起来并在此基础上执行操作。当你需要做一些基于多个服务调用的操作时，尝试一下适合你所选技术栈的响应式扩展，会发现它让你的代码变得非常简单。

4.2.9 微服务世界中的DRY和代码重用

DRY：Don’t Repeat Yourself。避免系统行为和知识的重复。

在微服务内部不要违反DRY，但在跨服务的情况下可以适当违反DRY。服务之间引入大量的耦合会比重复代码带来更糟糕的问题。

4.2.11 版本管理

语义化版本管理的每一个版本号都遵循这样的格式:MAJOR:MINOR:PATCH。MAJOR的改变意味着其中包含向后不兼容的修改；MINOR的改变意味着有新功能的增加，但应该是向后兼容的；PATCH的改变代表对已有功能的缺陷修复。

不想强迫客户端跟随服务端一起升级，可以在同一个服务上使老街口和新接口同时存在。

4.2.12 用户界面

4.2.13 与第三方集成

现实是一个组织对软件的需求不可能完全有内部满足，这时候就需要与第三方集成。然而与第三方集成并非易事。

对于一般规模的组织来说，如果某个软件非常特殊并且是你的战略性资产的话，就自己构建，如果不是就购买(使用第三方)。

任何定制化都只在自己可控的平台上进行，并限制工具的消费者的数量。

绞杀者模式：绞杀者可以捕获并拦截对老系统的调用，从而可以决定是把这些调用路由到老代码还是导向新代码，这种方式可以帮助我们逐步对老系统进行替换，从而避免影响过大的重写。

4.3 总结

集成的原则就是:

  • 无论如何避免数据库集成。
  • 理解REST和RPC之间的取舍，但总是使用REST作为请求/相应的开始。
  • 相比编排，优先使用协同。
  • 避免破坏性修改、理解Potel法则、使用容错性读取器。
  • 将用户界面视为一个组合层。

注：

Potel法则(鲁棒性原则):系统中的每个模块都应该宽进严出，对自己发送的东西要严格，对接受的东西则要宽容。

这里让你知道有哪些点需要学习以及正确的方向是什么，这对你的进一步学习很有帮助。
